范围与现状

目标是在 D:\sdez_165 对应的游戏程序集上，通过当前仓库的 MonoMod patch 实现 Mine。输入侧参考 MajdataEdit 和 MajSimaiX，运行语义与视觉侧参考 MajdataView，但最终行为必须适配游戏原生的 NoteData、Monitor 生命周期和 GameScoreList。

验收边界。 Q001-Q026 已全部转化为实现和离线验证项；代码、资源、部署与离线证据已经完成。用户明确要求助手不得自行启动游戏，因此 Q026 的实机矩阵和性能对照必须由用户运行并提供证据，当前不能标记为完整验收。

输入契约

已由代码确定的事实

主题 结论 设计影响
游戏判定域 NoteJudge.ETiming 共有 15 个有效结果；TooFast 与 TooLate 是两个 Miss 端点。 若采用结果反转，成功可映射为 Critical，失败映射为 TooLate。
统一结算入口 Tap、Hold、Touch、Slide 等最终都调用 GameScoreList.SetResult(index, kind, timing)。 结果反转可以集中实现，不必复制所有计分逻辑。
长物件语义 Hold/TouchHold 根据头判定、保持时长和最终状态合成一个普通结果；Slide/Wifi 根据路径完成度与结束时间生成普通结果。 “发生过接触”和“最终不是 Miss”并不严格等价。
Slide 数据粒度 MajSimaiX 区分 IsMine（星头）与 IsMineSlide（轨迹）；MA2 分别落在 STR 头记录与轨迹记录。 游戏侧需决定星头和轨迹是否保持可独立标记、独立结算。
Each MajdataView 会把 Mine 排除出普通 Each 表现；当前 patch 也会在 calcEach 后清理 Mine 关系。 这是现有一致行为，但仍需在后续确认是否是正式产品规则。
成绩链路 游戏从逐物件 JudgeResultSt 同时派生分数、DX Score、Combo、Miss、FC/AP、Life 和最终上传结果，没有独立 Mine 结果桶。 让 Mine 进入正式成绩可直接复用原链路；中立化则必须同步修改分母、理论 Combo、完成条件和上传兼容。
原版权重 Critical 时 Tap/Touch 为 500、Hold 为 1000、Slide 为 1500、Break 为 2500 加 100 奖励；DX Score 均为 3。 当前 patch 会自然沿用这些权重；若统一按 500 计，必须同时重算理论分母并移除 Mine Break 奖励。
组合修饰 MA2 的 Break/EX 记录可同时携带 !m；当前 patch 会保留基础计分类别，但用 Mine 灰色覆盖并抑制 Break/EX 的可见反馈。 需要明确“计分继承基础类型、表现由 Mine 主导”是否为正式优先级。
现有类型覆盖 参考实现与当前 patch 均已有 Tap/Star、Hold、Touch、TouchHold、Slide 和 Fan/Wifi 的解析、状态绑定、灰色覆盖及反馈抑制入口。 全类型支持无需新增另一套判定模型；主要剩余风险是逐类型实机回归。
连接 Slide 结果槽 目标游戏将内部 ConnectSlide 结果槽预置为 End，由首个可计分 Slide 对整条连接路径调用一次 SetResult。 若内部段独立结算，必须改写总数、理论分母和连接链生命周期；共享状态可直接沿用原生结构。
同头多 Slide MajSimaiX 将 * 后的轨迹解析为独立无头 Slide；目标游戏把同时间同起点的多条 Slide 挂到一个星头下，但每条轨迹仍有独立结果槽。 共享星头不要求共享轨迹 Mine 状态，分支可分别标记和结算。
AutoPlay 判定 原版 AutoPlay 可生成 Critical、Perfect、Great、Good、Miss 或 Random；当前 patch 在 Mine 的 NoteCheck 内临时切为 None，阻止自动按下、保持、触点和滑动。 阻止后 Mine 会自然得到原始 TooLate 并反转成功；不阻止则结果取决于 AutoPlay 模式。
输入来源 目标代码没有独立 DJAuto 状态；外部工具若通过 InputManager 注入事件，与实体按键和触摸无法区分。当前 patch 只关闭内建 AutoPlay，仍读取这些输入。 保留输入会让手动/DJAuto 触雷；若要免疫，必须在 AutoPlay 避雷期间额外屏蔽全部 Mine 输入。
Ghost 存储契约 UserGhost.CreateGhostBitData 直接保存正式 JudgeResultSt.Timing 的 4 bit 索引，回放数据已经包含 Mine 反转结果。 Ghost 写入、预载和实时回放必须绕过 Mine 转换，防止成功与失败发生二次反转。
Track Skip 顺序 原版 SetResult 在 IsTrackSkip 时强制把所有结果改为 TooLate；当前 patch 的转换发生在调用原版之前，因此最终仍是失败。 这是显式跳过的特殊规则；正常歌曲结束时的未触碰 Mine 仍按 Q001 反转成功。
非实时结果上下文 正常收尾给未判定物件补原始 TooLate；Quick Retry 不持久化结果；Ghost 与目标达成率分配保存或生成的是最终判定。 只有实时物件结果进入 Mine 转换；Ghost/目标成绩必须直通，Quick Retry 重建时清空 Mine 运行状态。
当前 Mine 尾字段识别 当前 patch 在 Note 的最终扩展字段中直接使用大小写敏感的 str.Contains("!m")；命中后临时移除全部精确 !m，其余修饰和顺序不变。 这是用户在 Q012 后追加的读取兼容性修订；Mine 不再解释 Soflan 或 !y，各补丁独立消费自己的 token。
无效 Mine 输入边界 规范化导出仍可限制重复和组合，但游戏端 Mine 读取器按用户修订接受重复 !m 与未知共存标记；错误位置、非 Note、连接 Slide 冲突和资源错误仍属于 Mine 拒绝边界。 非法 #groupFspeed 由 SoflanSupport 的正则解析器记录并抛出，避免 Mine 与 Soflan 重复解释同一语法。
错误日志能力 当前 PatchLog.WriteLine 带有 [Conditional("DEBUG")]，Release 会在调用点移除日志；现有后端可异步写入 dpMineSupport.log 并转发 Unity 日志。 无效 Mine 的拒绝日志必须使用不受 DEBUG 限制的 Error 入口，不能复用当前仅调试日志作为唯一诊断。
旧读取器边界 !m 不创建新 MA2 Note ID；不了解扩展的读取器可能忽略尾字段，并把记录解释为普通基础物件。 当前 patch 无法改变未安装 patch 的外部读取器行为，只能明确兼容政策并在受控工具链中提示能力要求。
Each 清理风险 当前 NormalizeEach 先清空 Mine 自身的 isEach/parent/eachChild，再从其他节点的 eachChild 移除 Mine；当 Mine 原本是 Each 根节点时，普通子节点可能仍保留指向 Mine 的 parent。 正式实现必须双向修复或重建剩余 Each 图，不能只清理 Mine 节点本身；普通节点之间的合法 Each 关系应保持一致。
Touch Chain 语义 目标游戏的 TouchChainList/TouchHoldChainList 按成员数的约 51% 判断链成功，并把 ChainJudge 传给尚未判定的其他触点；GameScoreList.SetResult 还会为每个 Touch 成员更新链状态。 Mine 若留在原生链中，就不再是单物件独立判定：Mine 的反转结果可能改变普通 Touch 的后续结算，反之亦然。
Soflan 速度机制 目标游戏的移动、出现时间和判定窗口由基础物件的 HS/SV/Soflan 路径计算；当前 Mine patch 只绑定 Mine 状态、视觉和结算，没有改写速度或时间字段。 Mine 与合法 #groupFspeed/FixedSoflan 的组合可以保持正交；解析顺序不应改变运动或判定时间。
当前视觉后端 MineVisual 把所有子 SpriteRenderer.color 的 RGB 设为 (0.68, 0.68, 0.68) 并保留 Alpha；等通道乘色只会压暗纹理，不会消除纹理原有色相。 当前实现不能可靠产生参考设计的灰白 Mine，尤其在红、黄、蓝等高饱和皮肤上仍会显色。
参考视觉资源 MajdataView 的 MineSpriteEffect shader 使用 Rec.709 亮度并映射到 0.58..1.0 灰阶，保留纹理轮廓与透明度；目标 MonoMod 项目当前不携带 shader/material 或 AssetBundle。 若采用真正灰白效果，部署产物必须增加与 patch 同版本的资源包和明确的装载/恢复生命周期。
对象池恢复 当前 MineVisual 只缓存 Renderer 的原始 Color，没有保存 sharedMaterial 或其他渲染状态；缓存还以 Renderer 强引用为键。 方案 A 必须对 Apply/Clear 做幂等快照与完整恢复，在结束、对象回池、Quick Retry 和异常清理时释放缓存，保证普通物件不继承 Mine 材质。
当前结果反馈 候选 patch 会在 Mine 的结束路径抑制 JudgeGrade、Fast/Late、普通判定音、Break/EX、Touch 与 Slide 成功特效，但没有创建 Mine 专用成功或失败反馈。 玩家触雷后主要只能从 Combo/Life/成绩变化察觉原因；若要即时因果反馈，需要新增独立于普通判定表现的 Mine 失败特效。
MISS 特效调用顺序 各物件原生 EndNote/Slide 结束路径会先用物件的原始判定初始化 JudgeGrade/SlideJudge，随后才调用 GameScoreList.SetResult 完成 Mine 结果反转。 Q020 不能通过简单恢复原特效实现；必须继续抑制原始 Critical/Perfect 等效果，再对实时触雷按最终 TooLate 主动播放一次对应物件族的原生 MISS 视觉。
反馈去重 同一运行时物件可能经过基类和派生类包装，Slide 连接链也包含多个内部段；直接在多个 EndNote/SetResult hook 中播放会重复触发 MISS 视觉或声音。 正式实现需要按逻辑结果索引和实时上下文集中调度 Q020/Q021，保证头/本体独立、连接链整链一次、非实时路径零次。
皮肤与对象池策略 原生 Initialize 先选取当前皮肤 Sprite/Material，Mine wrapper 随后应用效果；Hold 状态和动态 Slide 箭头可能在运行期间改变或新增 Renderer。 每次激活先清除旧快照，再捕获当前皮肤状态；运行期间覆盖动态 Renderer，回池/重试/异常时完整恢复，因而无需维护 Mine 专用皮肤表。
统计与上传结构 NoteScore.EScoreType 只有 Tap、Hold、Slide、Break、Touch、BreakBonus；UserPlaylog 也只上传这五类的 Critical/Perfect/Great/Good/Miss，没有 Mine 字段。 按基础类型归档可直接保持现有结果 UI 与服务器契约；新增官方 Mine 桶需要同步改结果界面、存档、上传 VO 和服务器协议。
判定显示设置 每位玩家已有 DispJudge、DispJudgePos 和 DispJudgeTouchPos；JudgeGrade.Initialize 在显示位置为 Off 时会直接返回。 通过原生入口播放 Q020 的 MISS 会自然遵循玩家设置；强制显示需要主动绕过现有偏好。
多玩家数据模型 目标游戏最多维护四个按 monitor 索引区分的 GameScoreList，并通过 NotesManager.Instance(monitorIndex) 提供各自 NoteData；Ghost 使用独立玩家槽与最终结果。 Mine 可按 (GameScoreList/monitor, noteIndex) 隔离，随后继续服从 Utage/Party 原有的 Chain、Combo 与排名聚合。
当前全局状态风险 候选 patch 的 RunNoteCheck 会临时改写静态 GameManager.AutoPlay，反馈抑制使用进程级 feedbackSuppressionDepth；二者都没有 monitor 或结果来源维度。 正式实现应使用按运行时物件、monitor 和来源作用域的上下文，并直接跳过 Mine 自动判定入口，避免一个玩家的 Mine 短暂影响另一玩家。
生命周期清理缺口 当前 ResetScore 只移除 MineScoreIndices；运行时物件字典、视觉快照和反馈去重状态依赖各对象正常走到 Release，异常/重试路径没有统一清理入口。 谱面初始化、Quick Retry、重载、加载失败和游戏退出都必须幂等清空瞬态状态，同时保留可复用且已验证的只读资源缓存。
当前性能热点 多个物件的 Execute 每帧调用 MineVisual.Apply，后者每次执行 GetComponentsInChildren<SpriteRenderer>(true) 并创建 Renderer 数组。 正式实现不能在稳定游戏帧重复扫描层级或产生托管分配；Renderer 快照应在初始化/动态创建点增量维护，非 Mine 路径保持近零开销。
验证状态 当前 patch 有 Debug/Release 编译与离线 MonoMod 合并记录，没有完整实机谱面回归证据。 最终方案必须附带实机验证矩阵，不能把“能合并”视为“行为正确”。

Q001 · 已确认 A · 根决策

Mine 的失败条件究竟由什么定义？

是沿用 MajdataView，把物件的“最终普通判定”二值反转；还是实现更严格的碰撞语义，只要在危险窗口内发生过任何有效按键、触摸、保持或滑动接触，就立即视为 Mine 失败？

方案 A：反转最终普通判定（推荐）

普通结果为 TooFast/TooLate 时 Mine 成功并记为 Critical；其余结果 Mine 失败并记为 TooLate。这与 MajdataView 一致，可复用每类物件成熟的判定窗口，并把转换集中在结算入口。

方案 B：任意有效接触即失败

独立记录按钮、触区、Hold 按压过程、Slide/Wifi 路径命中等触发状态。语义更像真正的地雷，但必须逐类定义危险窗口、跨帧去重、擦边和连接 Slide 传播，复杂度与回归范围明显增加。

用户结论：方案 A。 Mine 沿用基础物件的原生判定窗口与生命周期；原始 TooFast/TooLate 映射为成功 Critical，其他原始结果映射为失败 TooLate。不新增独立接触检测状态机。

Q002 · 已确认 A · 成绩模型

Mine 的反转结果是否进入游戏正式成绩链？

成功避开 Mine 与触发 Mine，是否应像普通 Critical/Miss 一样影响分数、DX Score、Combo、FC/AP、Life、达成率和最终上传结果？本问只决定是否进入正式成绩，具体沿用哪种物件权重将在下一问处理。

方案 A：进入正式成绩（推荐）

成功 Mine 作为 Critical 增加 Combo 并取得对应分值；失败 Mine 作为 TooLate 丢分、断 Combo，并参与 Miss、FC/AP、Life 与上传结果。它继续占用原 MA2 物件的结果槽。

方案 B：成绩中立

Mine 只提供玩法与视觉反馈，不影响原版成绩、Combo、Life 或称号。需要把它从原版分母和理论 Combo 中剔除，并新增独立完成计数，否则成绩与谱面总数会不一致。

用户结论：方案 A。 Mine 的反转结果进入正式成绩链，影响分数、DX Score、Combo、FC/AP、Life、达成率和最终上传结果。

Q003 · 已确认 A · 计分权重

Mine 应沿用基础物件权重，还是统一为固定权重？

在成功 Mine 统一记录为 Critical 的前提下，Hold、Slide、Break Mine 的正式分值和理论分母，是保留其原 MA2 类型，还是全部按一个 Tap 级 Mine 计算？

方案 A：沿用基础物件权重（推荐）

Tap/Touch Mine 为 500、Hold Mine 为 1000、Slide Mine 为 1500、Break Mine 为 2500 加 100 奖励，DX Critical 均为 3。Mine 只反转判定，不改变原记录的计分类别。

方案 B：统一 Mine 固定权重

对齐 MajdataView，所有 Mine 均按 500、DX Critical 3 计算，并屏蔽 Break 奖励。需要为 Mine 重算理论总分、分类统计与成绩分母，不能只修改 SetResult 的类型。

用户结论：方案 A。 Mine 保留原 MA2 基础物件的计分类别与理论权重；Tap/Touch、Hold、Slide、Break 分别沿用原版权重，Break Mine 也保留 Break 奖励。

Q004 · 已确认 A · 修饰优先级

Mine 与 Break/EX 组合时，玩家可见表现由谁主导？

同一记录同时带有 !m 和 Break/EX 类型时，是否保留 Q003 的基础计分权重，但让 Mine 覆盖其外观、音效、闪光、覆盖层与普通判定反馈？

方案 A：Mine 主导表现，基础类型负责计分（推荐）

接受 Mine + Break/EX。物件按 Mine 的灰色危险样式显示并抑制 Break/EX 专属反馈；原 Break/EX 类型仍提供原生判定窗口及 Q003 已确认的计分权重。

方案 B：叠加 Break/EX 表现

在 Mine 灰色基础上仍显示 Break 闪光、EX 覆盖层及相关音效。信息更完整，但会弱化 Mine 辨识度，也可能让触发失败看起来像普通成功命中。

方案 C：禁止组合

遇到 Break/EX 记录上的 !m 时拒绝谱面或忽略 Mine。规则最单纯，但会主动放弃 MajSimaiX 和 MA2 尾标记已经能够表达的组合。

用户结论：方案 A。 Mine 主导组合物件的外观、音效和特效，Break/EX 的玩家可见反馈被抑制；基础类型继续负责原生判定窗口和 Q003 已确认的计分权重。

Q005 · 已确认 A · 类型覆盖

哪些标准物件族允许使用 Mine？

是否让所有可独立结算的标准物件都接受 !m，包括 Tap/Star、Hold、Touch、TouchHold，以及 Slide/Fan/Wifi 本体？Slide 的星头、轨迹和连接段粒度将在下一问单独确定。

方案 A：全部标准物件族（推荐）

支持 Tap/Star、Hold、Touch、TouchHold、Slide 与 Fan/Wifi，并覆盖各自的 Break/EX 变体。每类继续产生原生最终判定，再按 Q001 统一反转。

方案 B：仅瞬时物件

只支持 Tap/Star/Touch，拒绝或忽略 Hold、TouchHold 和 Slide/Fan/Wifi 上的 !m。规则较窄，但会割裂 MajSimaiX 已支持的 Mine 语法。

方案 C：指定白名单

由你逐项指定允许的物件族。适合有明确玩法限制的情况，但编辑器、导出器、读取器和测试矩阵都需要维护同一份白名单。

用户结论：方案 A。 Mine 支持全部可独立结算的标准物件族：Tap/Star、Hold、Touch、TouchHold、Slide 与 Fan/Wifi，并覆盖各自的 Break/EX 变体。

Q006 · 已确认 A · Slide 粒度

Slide/Fan/Wifi 的星头和轨迹 Mine 标记是否独立？

谱面能否分别表达“仅星头是 Mine”“仅轨迹是 Mine”和“星头、轨迹都是 Mine”，并让头与轨迹继续占用各自原生结果槽独立结算？

方案 A：头与轨迹完全独立（推荐）

STR 星头只读取 IsMine，Slide/Fan/Wifi 轨迹只读取 IsMineSlide。两者都标记时产生两个独立反转结果；无头 Slide 只能标记和结算轨迹。

方案 B：头与轨迹强制联动

任一侧出现 Mine 标记就传播到另一侧，星头和轨迹同时成为 Mine，并仍各自结算。规则更整体，但无法表达 MajSimaiX 已支持的仅头或仅轨迹 Mine。

用户结论：方案 A。 Slide/Fan/Wifi 的星头与轨迹 Mine 标记完全独立；仅头、仅轨迹和两者同时为 Mine 均有效，两者同时标记时占用各自原生结果槽独立结算。

Q007 · 已确认 A · 连接 Slide

连接 Slide 的多个内部轨迹段是否共享一个 Mine 本体？

一条连接 Slide 在 MA2 中拆成首段和若干 ConnectSlide 记录后，是否把整条轨迹视为一个 Mine 状态，所有段统一显示，并只在原生首段结果槽结算一次？混合尾标记如何校验将在后续输入规则中确定。

方案 A：整条连接链共享，单次结算（推荐）

所有内部段共享一个 IsMineSlide 语义和 Mine 外观；连接段仅参与路径判定，不生成额外成绩，整条链最终只反转一次 Slide 结果。

方案 B：各段独立 Mine

每个 MA2 轨迹记录可独立标记和结算 Mine。需要解除内部段的 End 结果、增加理论总数，并为每段定义局部判定窗口，改变原生连接 Slide 玩法。

用户结论：方案 A。 一条连接 Slide 的所有内部轨迹段共享一个 Mine 本体状态和统一外观；内部 ConnectSlide 不产生额外成绩，整条链只反转一次原生 Slide 结果。

Q008 · 已确认 A · 同头多 Slide

同一星头下的多条 Slide 轨迹是否分别拥有 Mine 状态？

使用 * 形成同头多 Slide 时，是否允许一条轨迹为 Mine、另一条保持普通，并让每条轨迹按自己的 IsMineSlide 独立显示和结算？共享星头本身继续由 Q006 的 IsMine 单独控制。

方案 A：各轨迹分支独立（推荐）

每条同头轨迹分别读取自己的 Mine 标记和原生结果槽。Mine 与普通轨迹可以共享一个星头；多条轨迹都为 Mine 时，各自产生一次反转结果。

方案 B：同头轨迹强制联动

任一轨迹标记 Mine，就把同一星头下所有轨迹都设为 Mine。整体视觉更统一，但无法表达 MajSimaiX 当前支持的分支级 Mine。

用户结论：方案 A。 同一星头下的各条 Slide 轨迹分别拥有 Mine 状态和结果槽，Mine 与普通轨迹可以并存；共享星头本身继续由独立的 IsMine 控制。

Q009 · 已确认 A · AutoPlay

普通 AutoPlay 是否应自动避开所有 Mine？

启用任何 AutoPlay 判定模式时，Mine 是否跳过自动按下、自动保持、自动触点和自动滑动，等待原生窗口自然超时并按 Q001 反转为成功？普通物件仍照常使用所选 AutoPlay 判定。

方案 A：AutoPlay 主动避雷（推荐）

Mine 在判定期间临时禁用 AutoPlay，所有类型自然超时为原始 Miss，再记录为成功 Critical；普通物件不受影响。这代表自动演奏采用最优 Mine 策略。

方案 B：照常生成原始自动判定

Mine 也执行 AutoJudge 后再反转：Critical/Perfect/Great/Good 模式会触雷失败，Miss 模式成功，Random 模式随机成败。更适合判定调试，但不再是最优自动演奏。

用户结论：方案 A。 内建 AutoPlay 主动避开所有 Mine，不自动按下、保持、触摸或滑动；Mine 按原生窗口自然超时并反转为成功，普通物件仍使用所选 AutoPlay 判定。

Q010 · 已确认 A · 输入边界

AutoPlay 避雷期间，真实输入是否仍可触发 Mine？

内建 AutoPlay 被 Mine 临时抑制时，是否继续读取实体按键、触摸和经 InputManager 注入的 DJAuto 事件，并在它们命中 Mine 时按 Q001 判为失败？

方案 A：真实输入仍可触雷（推荐）

只屏蔽内建自动判定，不屏蔽输入系统。玩家或 DJAuto 实际命中 Mine 时照常生成原始非 Miss，并反转为失败；没有输入时 AutoPlay 才会成功避开。

方案 B：AutoPlay 下 Mine 完全免疫

AutoPlay 启用期间忽略 Mine 的全部输入，保证其始终成功。需要额外屏蔽 InputManager 事件；未设置 AutoPlay 标志的外部 DJAuto 仍无法被可靠识别。

用户结论：方案 A。 AutoPlay 只屏蔽内建自动判定；实体输入、触摸及经 InputManager 注入的 DJAuto 事件仍可命中 Mine，并按 Q001 反转为失败。

Q011 · 已确认 A · Track Skip

显式 Track Skip 后，剩余 Mine 应记为失败还是成功？

玩家主动跳过歌曲剩余部分时，尚未结算的 Mine 是否与其他剩余物件一样强制记为 TooLate 失败，而不是因为“没有触碰”获得成功 Critical？

方案 A：Track Skip 强制失败（推荐）

所有未结算 Mine 最终保持 TooLate，不取得剩余分数，并计入 Miss/Combo/Life 结果。正常播放至歌曲结束但没有触碰 Mine，仍按 Q001 成功。

方案 B：Track Skip 也视为成功避开

剩余 Mine 在跳过时映射为 Critical 并取得分数。需要让 Mine 转换覆盖原版 Track Skip 的强制 Miss，可能出现主动跳过却增加成绩的结果。

用户结论：方案 A。 显式 Track Skip 后，所有未结算 Mine 强制保持 TooLate 失败，不取得剩余分数；正常播放至歌曲结束但未触碰 Mine 仍按 Q001 成功。

Q012 · 后续修订为 Contains 兼容读取 · MA2 输入

MA2 的 Mine 尾标记采用严格语法还是宽松搜索？

读取器是否只在物件记录的修饰尾字段中识别一次精确小写 !m，并允许它与一个合法 #groupFspeed 块按任意先后顺序共存？

方案 A：结构化严格语法（推荐）

接受 !m、!m#12F600、#12F600!m 等契约形式；只认精确小写标记和合法尾字段，不在其他列或任意字符串中搜索子串。

方案 B：维持宽松搜索

扫描所有字段，只要压缩空白后类似包含或结尾为 !m 就视为 Mine。兼容随意写法，但可能把未知扩展、注释或损坏字段误判为 Mine。

用户后续修订（覆盖原方案 A 的读取严格度）：游戏端在 Note 最终扩展字段中直接用大小写敏感的 str.Contains("!m") 判断 Mine，并移除全部精确 !m 后把其余内容原样交给其他补丁；因此 #1!m!y、!y!m#1 与重复 !m 都按 Mine 读取。规范化导出规则不因此改变。

Q013 · 保留结构错误日志，尾修饰语法改由各消费者负责 · 错误边界

遇到结构无效的 Mine 数据时，读取器应如何处理？

用户后续修订后，重复 !m、未知私有标记和混合排列不再由 Mine 判为错误；Mine 仍拒绝错误字段位置、非 Note 标记和连接 Slide 链状态冲突，SoflanSupport 独立拒绝无效或重复的 #groupFspeed。

方案 A：拒绝受影响谱面并精确报错（推荐）

停止加载当前谱面或难度并返回失败，但不让整个进程崩溃；日志包含 MA2 记录序号、记录类型、原始尾字段和具体原因，要求作者修正数据后再加载。

方案 B：警告后降级为普通物件

丢弃无效 Mine 修饰并继续加载基础物件。可用性较高，但会把本应危险的物件静默变成普通物件，使谱面行为和作者意图不一致。

方案 C：尽力自动修复

自动去重、重排尾块或把连接链统一成某一段的状态。能容忍部分旧数据，但修复方向存在歧义，且编辑器、预览器和游戏可能各自得到不同结果。

用户结论保留，并按后续修订缩小适用范围：Mine 结构错误仍拒绝当前谱面或难度，Debug/Release 均写可定位 Error；尾字段中其他 token 的语法由对应补丁负责，Mine 不因 !y、重复 !m 或 Soflan 内容本身拒绝。

Q014 · 已确认 A · 兼容政策

不了解 !m 的旧 MA2 读取器，应被视为受支持环境吗？

由于 Mine 沿用基础 Note ID，旧读取器可能忽略私有尾字段并把 Mine 当成普通物件；当前游戏 patch 无法替未安装 patch 的外部读取器阻止这种行为。

方案 A：明确要求 Mine-aware 读取器（推荐）

把含 !m 的谱面定义为 Majdata 扩展内容，只有明确支持该扩展的游戏、编辑器和预览器才属于受支持环境。旧读取器降级为普通物件属于不受支持行为，文档、导出提示和兼容测试都应明确这一点。

方案 B：正式支持降级为普通物件

允许旧读取器忽略 !m，并把普通物件播放视为预期兼容结果。谱面通常仍能打开，但玩法、成绩、Combo 和生存结果都会与 Mine-aware 环境不同。

方案 C：引入强制拒绝旧读取器的格式门槛

增加新记录类型、版本头或其他破坏兼容的标记，让旧工具更可能直接拒绝谱面。这会改变 Q012 已确定的“不新增 MA2 Note ID”契约，并扩大编辑器、预览器和导出器改造范围。

用户结论：方案 A。 含 !m 的谱面要求 Mine-aware 读取器；旧读取器把 Mine 当普通物件属于不受支持行为。保留现有私有尾字段契约，不新增 MA2 Note ID 或强制格式门槛。

Q015 · 已确认 A · Each 关系

Mine 是否参与普通 Each/EachLine 关系？

这里先只决定普通同时按下物件的 Each 连接和引导线；Touch/TouchHold 的触点组归属另列为下一问。若同一时刻有 Mine 与普通物件，是否让 Mine 保持独立，同时让剩余普通物件继续组成合法 Each？

方案 A：Mine 排除出 Each（推荐）

Mine 不设置 isEach、parent、eachChild 或 indexEach，不显示普通 EachLine；同一时刻的普通物件在移除 Mine 后仍按原规则互相组成 Each。若剩余普通物件不足以成组，则不显示引导线。

方案 B：Mine 参与普通 Each

保留原生 Each 图和引导线，让 Mine 与普通物件看起来属于同一组，但每个 Mine 仍按 Q001 独立反转结果。视觉关系更完整，却容易暗示 Mine 与普通物件共享成功条件。

方案 C：Mine 使用独立 Each 组

不与普通物件连线，但把同时出现的多个 Mine 组成专用 Mine 引导线或组状态。需要新增表现、对象生命周期和输入语义，目前参考实现没有这种概念。

用户结论：方案 A。 Mine 排除出普通 Each/EachLine；移除 Mine 后，剩余普通物件继续按原规则组成 Each。实现必须双向修复或重建剩余 Each 图，避免普通节点留下指向 Mine 的悬挂关系。

Q016 · 已确认 A · Touch Group

Mine Touch/TouchHold 是否参与原生触点组？

目标游戏的 TouchChain/TouchHoldChain 会按组内成员比例判定，并将链结果推送给尚未结算的成员。若同一触点组混有 Mine 与普通 Touch，是否让 Mine 影响普通成员的链式结果？

方案 A：Mine 排除出触点组（推荐）

Mine Touch/TouchHold 保持 indexTouchGroup = -1，不进入 TouchChainList 或 TouchHoldChainList，独立走 Q001 判定和正式结算；过滤 Mine 后，剩余普通触点重新组成原生链。

方案 B：Mine 参与原生触点组

保留 Mine 在原生链中的成员资格，让 51% 阈值和 ChainJudge 作用于所有成员。这样一次链式结果可能同时结算 Mine 与普通触点，Mine 的反转顺序必须额外定义。

方案 C：建立独立 Mine 触点组

普通触点链完全不受 Mine 影响，同时为同组 Mine 另建专用链。需要重新定义 Mine 链的阈值、输入传播、TouchHold 保持过程和统计粒度，超出当前原生模型。

用户结论：方案 A。 Mine Touch/TouchHold 不加入原生触点链，独立走 Q001 判定和正式结算；过滤 Mine 后，普通触点重建各自的原生 Touch/TouchHold 链。

Q017 · 已确认 A · Soflan 优先级

Mine 与 Soflan/HS/SV、FixedSoflan 同时出现时，谁控制时间机制？

Q012 已确定 !m 可以和一个合法 #groupFspeed 块共存；现在只决定运行时优先级：速度修饰是否继续控制物件的移动、出现时间和原生判定窗口？

方案 A：速度修饰与 Mine 正交（推荐）

完整保留基础物件的 Soflan/HS/SV/FixedSoflan 运动和时间轴，Mine 只负责灰白表现、输入避让和 Q001 结果反转。!m#12、#12!m 等等价尾字段必须得到相同的时间行为。

方案 B：Mine 覆盖速度修饰

Mine 使用固定的默认速度或专用危险速度，忽略记录中的 Soflan/FixedSoflan。视觉和判定更容易统一，但会改变作者为该物件设置的出现节奏与判定窗口。

方案 C：禁止速度修饰与 Mine 组合

遇到 Mine 与 #groupFspeed 共存时拒绝谱面或忽略 Mine。实现边界简单，但会否定 Q012 已确认的组合语法并割裂编辑器与游戏行为。

用户结论：方案 A。 Soflan/HS/SV/FixedSoflan 完整控制基础物件的移动、出现时间和原生判定窗口；Mine 只负责表现、输入避让与结果反转，等价尾字段不得产生时序差异。

Q018 · 已确认 A · 视觉后端

Mine 的灰白外观采用哪种渲染方案？

当前等通道 tint 不能真正去色。是否随 patch 部署专用 shader/material 资源，对所有 Mine Sprite 做亮度灰阶映射，并在对象复用时恢复原材质与颜色？

方案 A：专用灰阶 shader/material（推荐）

随 patch 提供配套 AssetBundle，采用参考实现的 Rec.709 亮度与 0.58..1.0 灰阶映射；覆盖 Tap/Star、Hold、Touch/TouchHold、Slide/Fan/Wifi 的全部子 Sprite，并保存、恢复每个 Renderer 原材质、颜色和 Alpha。

方案 B：保留代码内简单 tint

不增加资源包，继续把 Renderer 颜色乘以统一灰值。部署最简单，但彩色纹理仍保留色相，无法保证 Mine 在不同皮肤上有一致灰白辨识度。

方案 C：为每类物件维护 Mine Sprite

提供独立的 Mine Tap、Hold、Touch、Slide 等纹理并逐类替换。可完全控制造型，但需要覆盖所有皮肤、Break/EX、动态 Slide Bar 和后续皮肤变体，维护面最大。

用户结论：方案 A。 随 patch 部署专用灰阶 shader/material AssetBundle，采用 Rec.709 亮度映射；复用基础皮肤轮廓和动画，并在物件结束或回池时完整恢复原材质、颜色、Alpha 与缓存状态。

Q019 · 已确认 A · 资源失败

Mine 视觉资源不可用时，含 Mine 的谱面是否继续播放？

不可用包括 AssetBundle 缺失或版本不匹配、目标 Material 不存在、shader 加载失败/不受支持。该错误可以在谱面进入游戏前验证，且不需要影响不含 Mine 的普通谱面。

方案 A：仅拒绝含 Mine 的谱面（推荐）

Mine 资源验证失败时不开始当前谱面或难度，干净返回加载失败；普通谱面继续可用。Debug/Release 都写一次 Error 日志，包含资源路径、patch/资源版本、失败阶段和异常信息，避免逐物件刷屏。

方案 B：降级为简单 tint 后继续

记录 Warning，并回退到当前的等通道乘色。谱面仍可游玩，但不同皮肤可能保留明显色相，Mine 与普通物件的辨识不再可靠。

方案 C：保留普通外观，仅运行 Mine 逻辑

不应用任何专用视觉，仍执行 Mine 判定与计分。危险物件看起来与普通物件相同，属于不可见玩法差异，最容易造成误判。

用户结论：方案 A。 AssetBundle、材质、shader 或版本验证失败时，只拒绝含 Mine 的当前谱面/难度；普通谱面继续可用。Debug/Release 均写一次包含资源路径、版本、失败阶段和异常信息的 Error 日志，不逐物件刷屏。

Q020 · 已确认 A（用户修正） · 结果反馈

Mine 成功避开与实时触雷失败，应如何显示？

普通 Judge、Fast/Late、Break/EX、Touch 与 Slide 成功特效继续按 Q004 被抑制；这里仅决定是否添加 Mine 自己的结果反馈。音效在下一问单独决定。

方案 A：仅实时触雷显示原生 MISS（用户修正后采用）

玩家或 DJAuto 的实时输入导致 Mine 最终失败时，按转换后的 TooLate 播放一次对应物件族的原生 MISS 视觉；成功避开时静默结束，不显示 Critical/Perfect 或 Fast/Late。Slide 头/本体按 Q006 各自一次，连接 Slide 本体按 Q007 整链一次；Track Skip、Ghost 和非实时成绩生成不批量播放。

方案 B：成功和失败都静默

维持参考实现与当前 patch 的反馈抑制，不新增专用特效。画面最干净，但玩家触雷时只能从 Combo、Life 或成绩变化推断原因。

方案 C：复用普通判定特效

成功显示 Critical/Perfect，失败显示 Miss，并恢复基础物件的判定动画。实现资源成本较低，但容易把“避开危险”表现成普通命中，并重新暴露 Q004 已抑制的 Break/EX 成功反馈。

用户结论：方案 A，并修正表现。 实时触雷显示游戏原生 MISS 视觉特效，未触雷成功则静默结束；原始非 Miss 特效必须被抑制，实际展示依据 Mine 转换后的最终 TooLate。Track Skip、Ghost 与其他非实时结算不播放反馈。

Q021 · 已确认 A · 音效

实时触雷显示 MISS 时，应播放什么声音？

成功避开继续完全静默；这里仅决定玩家或 DJAuto 实时触雷失败的声音。Break/EX、TouchHold 循环音、Slide 触摸音等基础命中声音仍按 Q004 被抑制。

方案 A：同步原生 MISS 判定音（推荐）

每个实时触雷逻辑结果播放一次游戏原生 MISS cue，与 Q020 的 MISS 视觉同步，并遵循玩家现有音效设置；不播放原始 Critical/Perfect、Break/EX、Hold Loop 或 Slide 命中音。Track Skip、Ghost 和成功避开不发声。

方案 B：使用专用 Mine 触雷音

在 AssetBundle 或配套音频资源中加入独立爆炸/警报 cue。辨识度最高，但需要额外处理音频格式、混音、音量、并发限制和资源版本。

方案 C：触雷也保持静音

只显示 Q020 的 MISS 视觉，不产生任何 Mine 相关声音。对安静游玩最友好，但快速谱面中视觉可能被其他物件遮挡。

用户结论：方案 A。 每个实时触雷逻辑结果同步播放一次对应物件族的原生 MISS 判定音，并遵循现有音效设置；成功避开、Track Skip、Ghost 和其他非实时路径保持静音，不恢复基础命中音。

Q022 · 已确认 A · 无障碍辨识

Mine 是否需要灰阶之外的非颜色视觉线索？

灰阶效果在彩色皮肤上明显，但低饱和/黑白皮肤本身可能与 Mine 接近。是否在专用 shader 中增加不依赖色相的第二线索？

方案 A：增加静态高对比轮廓/纹理（推荐）

在灰阶基础上加入稳定的明暗边缘或斜纹，不闪烁、不依赖红绿差异，并覆盖 Tap/Star、Hold、Touch/TouchHold 与 Slide/Fan/Wifi 条体；保持基础轮廓仍可识别。

方案 B：只使用灰阶去色

严格沿用参考视觉，不增加其他标记。画面最简洁，但在低饱和皮肤、低亮度显示器或复杂背景上，Mine 与普通物件可能不够容易区分。

方案 C：使用动态亮度脉冲

让 Mine 周期性闪烁或呼吸，辨识度最高且适用于所有物件形状，但快速谱面中更干扰注意力，也需要考虑闪烁与动态敏感用户。

用户结论：方案 A。 在灰阶 shader 中增加静态高对比轮廓或纹理，覆盖所有标准物件族；不依赖色相、不闪烁，并保留基础物件轮廓的可识别性。

Q023 · 已确认 A · 统计归属

是否增加独立的 Mine 成功/触雷统计？

Q002/Q003 已确定 Mine 的最终结果进入正式成绩并沿用基础物件权重；这里仅决定结果页、存档和上传是否还要出现独立 Mine 桶。

方案 A：不增加独立 Mine 统计（推荐）

Mine 最终 Critical/Miss 继续计入其基础 Tap/Hold/Slide/Break/Touch 类别，沿用现有成绩、结果页、FC/AP、Ghost 和 UserPlaylog；运行时只保留必要的内部 Mine 标记与诊断日志，不显示独立成功率。

方案 B：增加仅本地 Mine 统计

额外显示“避开数/触雷数/总数”，但不参与分数且不上报服务器。需要改本地结果 UI 和存档/重放展示，并明确该值与基础类型判定数重复计数。

方案 C：增加完整官方 Mine 桶

扩展 NoteScore、结果 UI、存档、Ghost 和上传协议，让 Mine 成为第六类官方统计。信息最完整，但会破坏现有数据契约，并需要服务器端同步支持。

用户结论：方案 A。 不增加独立 Mine 统计；Mine 最终结果继续按基础 Tap/Hold/Slide/Break/Touch 类型进入成绩、结果页、FC/AP、Ghost 和上传协议，仅保留内部标记与诊断信息。

Q024 · 已确认 A · 用户设置

触雷 MISS 视觉是否尊重玩家的原生判定显示设置？

Q022 的灰阶与静态轮廓属于物件身份，始终显示；这里仅决定触雷后的原生 MISS 结果文字/动画是否受 DispJudge 和位置设置控制。

方案 A：完全尊重现有判定显示设置（推荐）

通过各物件族原生 MISS 入口播放，沿用每位玩家的判定样式和 Tap/Touch 显示位置；玩家关闭相关判定显示时不强制出现 MISS。Q021 的声音仍独立遵循现有音效设置。

方案 B：触雷 MISS 始终强制显示

即使玩家关闭原生判定显示，也绕过 Off 检查显示 MISS。危险反馈最明确，但破坏玩家的判定显示偏好，并可能在简洁 UI 配置中突然出现结果文字。

方案 C：新增独立 Mine 反馈开关

为触雷 MISS 增加单独的 On/Off 设置。控制最细，但需要扩展选项 UI、用户配置序列化、默认值和旧存档迁移。

用户结论：方案 A。 触雷 MISS 视觉完全遵循每位玩家的 DispJudge 与 Tap/Touch 显示位置；关闭判定显示时不强制出现。Mine 本体的灰阶和静态高对比标记仍始终可见。

Q025 · 已确认 A · 多玩家

Mine 的判定与反馈是否按玩家完全隔离？

这里的“隔离”发生在游戏原有模式聚合之前：每个玩家先独立产生 Mine 最终结果，之后 Utage/Party 仍可按原规则计算共享 Chain、强制 Combo 或排名。

方案 A：每位玩家独立（推荐）

同一个谱面 Mine 可以由 P1 成功避开、P2 触雷失败；输入、AutoPlay 避让、分数、Life、MISS 视觉/音效和用户设置都按 monitor 隔离。Ghost 只提供自身最终成绩，不触发真人玩家反馈；原生多人聚合在这些最终结果之后照常运行。

方案 B：本地双人共享 Mine 结果

一名本地玩家触雷或避开后，另一名玩家同索引 Mine 使用相同结果。需要跨 monitor 同步判定与去重，并会让未输入的玩家被动得分或失败。

方案 C：Party 全员共享 Mine 结果

将 Mine 变成房间级共同危险，任一成员触雷影响所有人。需要新增网络消息、时序一致性、掉线与重连规则，目标协议当前没有该语义。

用户结论：方案 A。 每位玩家独立产生 Mine 输入、判定、成绩、Life 与反馈，再交给 Utage/Party 原有规则聚合；Ghost 使用自身最终结果且不触发真人反馈。实现不得通过全局 AutoPlay 或反馈状态跨 monitor 污染。

Q026 · 已确认 A · 最终验收

达到什么验证深度，Mine 才能标记为“实现完成”？

当前第一版已能编译和离线合并，但已发现严格解析、Ghost 二次反转、连接 Slide、Touch Group、全局状态、视觉资源与性能等实质缺口。

方案 A：完整证据链（推荐）

要求 Debug/Release 零错误构建、严格 MA2/结果上下文自动化测试、资源版本与失败测试、MonoMod 离线合并及 IL 核查，并完成实机矩阵：普通谱面零回归、全物件族与组合、Slide 粒度、手动/AutoPlay/DJAuto、Ghost、Track Skip、Quick Retry、1P/2P、皮肤/对象池、MISS 反馈和错误日志。密集谱面还需确认暖机后无 Mine 每帧分配或层级扫描，并与普通基线比较帧耗时。

方案 B：自动化与离线合并即可

完成构建、单元测试、资源检查和 MonoMod 合并后即标记完成，实机全类型/双人/性能仅列为后续验证。速度较快，但 Unity 对象层级、音效、显示设置和对象池问题无法由离线测试证明。

方案 C：编译加单次实机冒烟

只要求 Release 构建和一张简单 Tap Mine 谱面可运行。无法覆盖本轮已识别的跨类型、Ghost、连接链、资源失败和多人风险。

用户结论：方案 A。 Mine 只有在完整构建、自动化、资源失败测试、MonoMod 合并/IL 核查、全类型实机矩阵和性能对照形成证据链后才能标记完成。本轮 Q001-Q026 产品设计至此冻结。
